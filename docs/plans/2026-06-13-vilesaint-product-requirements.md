# VileSaint 产品需求文档（PRD）

版本：1.1  
域名：`vilesaint.com`  
产品定位：全球足球球迷的实时互联网审判庭  
上线范围：完整版本一次上线，不拆分 MVP / 二期  
产品语言：18 种首发语言，架构支持继续扩展  

---

## 1. 产品概述

### 1.1 一句话定义

VileSaint 是一个围绕全球足球比赛实时生成争议话题、英雄与反派投票、球迷人格和国家立场榜单的互动网站。

用户进入网站后，不需要先注册，也不需要学习规则，只需要对当前比赛发生的事件快速选择立场。投票后立即看到全球、国家和不同语言用户的观点差异，并生成带大二维码的分享卡，将新用户带回对应投票页面。

### 1.2 品牌主张

英文：

> Every match creates a saint and a villain.

中文：

> 每场比赛，都会诞生一个圣徒和一个恶人。

网站功能描述：

> The internet's court for football.

中文：

> 足球世界的互联网审判庭。

### 1.3 核心目标

1. 用户第一次进入后 10 秒内理解产品。
2. 用户在 30 秒内完成第一次投票并看到结果。
3. 每场比赛都能自动产生值得讨论和分享的新内容。
4. 分享结果本身具有身份表达价值，并通过二维码形成回流。
5. 不依赖广告投放，通过实时事件、自然搜索、群聊和社交分享增长。
6. 内容、翻译、投票生成、结果摘要和分享文案全部自动化。
7. 自动识别具有爆发潜力的比赛事件，生成适合不同平台和语言的社交内容并自动发布。
8. 产品必须好玩、有冲突、有记忆点，但不依赖辱骂、造谣或民族攻击制造热度。

### 1.4 非目标

- 不做普通比分网站。
- 不做体育新闻门户。
- 不做现金竞猜、博彩或赔率推荐。
- 不提供比赛视频、盗播链接或未经授权的比赛画面。
- 不使用人工后台逐条创建投票或翻译内容。
- 首次上线不考虑广告系统。
- 不把相同文案机械翻译后群发到所有语言和平台。
- 不使用自动账号在第三方社区批量灌水或绕过平台规则。

## 2. 产品原则

### 2.1 一眼就会用

首屏永远展示一个明确问题和两个至四个大选项。核心页面不出现长说明、复杂导航或必须填写的表单。

### 2.2 先参与，后看结果

投票前不展示完整比例，避免从众效应。投票后立即揭晓：

- 全球结果
- 用户所在国家或地区结果
- 四个语言群体结果
- 用户属于多数派还是少数派

### 2.3 先使用，后登录

匿名用户可以浏览、投票和分享。以下行为才提示创建账户：

- 保存长期人格和历史战绩
- 创建好友审判室
- 跨设备同步
- 进入长期排行榜

支持 Google、Apple 和邮箱 Magic Link。不要使用密码注册作为首选。

### 2.4 每个结果都能传播

所有投票结果、人格结果、国家榜和好友榜都必须能生成适合社交平台的分享卡，并带有清晰的大二维码和短链接。

### 2.5 AI 自动化但不能自动失控

AI 负责生成内容，不代表所有生成内容都直接发布。系统需要基于数据可信度、内容安全、重复度和争议价值进行自动校验。低置信度内容使用模板或不发布，不允许编造比赛事实。

## 3. 用户与使用场景

### 3.1 核心用户

#### A. 实时观赛用户

正在看比赛，希望立即表达对进球、红牌、点球、换人和裁判决定的立场。

#### B. 社交型球迷

不一定看完整比赛，但喜欢看争议、榜单、梗和朋友观点。

#### C. 国家队死忠用户

强烈认同某支国家队，愿意参与国家之间的比较和立场对抗。

#### D. 群聊组织者

希望在微信群、WhatsApp、LINE、KakaoTalk、Discord 或公司群中组织投票和排行榜。

#### E. 内容创作者

需要快速获得可分享的数据、争议结果和粉丝互动链接。

### 3.2 核心使用时刻

- 赛前：谁会赢、谁压力最大、谁会成为英雄或反派。
- 赛中：点球、红牌、争议判罚、错失机会、换人和关键进球。
- 赛后：本场 Saint、Villain、最被冤枉的人、谁该背锅。
- 每日：当天最大英雄、最大反派、最有争议事件。
- 淘汰后：责任审判、球迷情绪、国家队总结。
- 整届赛事：球迷人格、国家立场榜、最终 Saint/Villain 榜。

## 4. 信息架构

移动端底部使用 4 个主入口，桌面端使用顶部导航。

1. **Court / 审判**
   - 默认首页
   - 当前正在进行和最近结束的比赛
   - 实时投票事件流

2. **Matches / 比赛**
   - 今日
   - 赛程
   - 已结束
   - 每场比赛详情和全部审判

3. **Rankings / 榜单**
   - Saint 榜
   - Villain 榜
   - 国家立场榜
   - 最具争议事件

4. **Me / 我的**
   - 球迷人格
   - 投票历史
   - 好友审判室
   - 语言与国家设置

全站固定提供：

- 左上角 VileSaint 标识
- 右上角语言切换
- 当前比赛快捷入口
- 分享按钮

### 4.1 爆点导航状态

当系统发现重大争议时，导航出现一个短时状态入口：

- `🔥 WORLD DIVIDED`：全球观点接近五五开
- `⚖️ VERDICT FLIPPED`：多数意见发生反转
- `😇 NEW SAINT`：新球员登顶 Saint 榜
- `😈 NEW VILLAIN`：新球员登顶 Villain 榜
- `🌍 NATION CLASH`：两个国家用户观点严重对立

状态只在满足数据阈值时出现，不能由 AI 凭感觉制造。

## 5. 页面需求

## 5.1 首页：Live Court

### 页面目标

让第一次访问的用户立刻完成一次投票。

### 页面结构

1. 顶部极简导航
2. 当前赛事状态条
   - LIVE：显示正在进行的比赛
   - NEXT：没有直播时显示下一场倒计时
3. 主审判卡
   - 比赛双方、国旗、比分和比赛时间
   - 一个明确问题
   - 2 至 4 个大选项
   - 参与人数
   - 投票截止状态
4. 投票后结果动画
5. “分享我的判决”主按钮
6. 下一个审判
7. 今日热门审判
8. 每日 Saint / Villain 榜

### 首屏文案示例

问题：

> Was that really a penalty?

选项：

- Clear penalty
- Never a penalty
- Too close to call

中文：

> 这真的是点球吗？

- 明确是点球
- 绝对不是点球
- 很难判断

### 交互要求

- 点击选项立即投票，不增加确认步骤。
- 投票按钮高度不低于 56px。
- 投票后 300 至 500ms 内显示本地乐观结果。
- 用户可以撤回或修改一次，修改入口弱化展示。
- 向上滑或点击“下一条”进入下一个审判。

## 5.3 审判详情页

URL：`/court/{poll-slug}`

这是最重要的分享落地页。

### 投票前

- 比赛上下文
- 问题
- 选项
- 截止时间
- 不展示完整投票比例

### 投票后

- 全球投票比例和人数
- 用户所在国家或地区结果
- 中文、英文、日文、韩文用户结果
- “你属于全球 X% 的少数派/多数派”
- AI 生成的中立结果摘要
- 分享卡预览
- 大二维码
- 下一个相关审判

### 结果摘要示例

> 全球多数用户认为这不是点球，但巴西和日本用户的判断明显相反。目前分歧仍在扩大。

摘要只能描述聚合结果，不评价用户智力、民族或人格。

## 5.4 Saint / Villain 赛后评选

每场比赛结束后自动生成两个核心问题：

1. Who was the Saint of the match?
2. Who was the Villain of the match?

候选人由比赛事件、官方阵容和统计数据自动产生：

- 关键进球或助攻球员
- 门将
- 红牌球员
- 错失点球或重大机会球员
- 主教练
- 裁判（只有存在明确判罚事件时）

每个候选人显示：

- 姓名
- 所属队
- 关键事实标签

不直接显示未经授权的球员照片。默认使用国旗、号码、姓名首字母和抽象头像。

- 多数派 / 逆风派
- 忠诚 / 中立

- The Loyal Saint / 忠诚圣徒

- 与全球用户的相似度
- 可分享人格卡和大二维码

至少完成 5 次投票后生成人格，少于 5 次时展示进度。

系统架构不能将语言写死。新增语言应只需要增加 locale 配置、固定词库、实体映射和 AI 质量规则。

### 入口设计

桌面端右上角只显示地球图标和当前语言，例如 `🌐 English`。点击后打开语言面板。

移动端右上角显示地球图标和当前语言缩写，例如 `EN`。

语言面板包括：

- 搜索框
- 推荐语言
- 全部语言
- 每种语言使用原生名称

推荐语言最多显示 4 个，根据浏览器语言、用户地区、当前比赛双方和历史选择动态排序。不要把 18 个选项平铺在导航栏。

### 默认语言判断顺序

1. 用户手动选择并保存的语言
2. URL 语言
3. 浏览器语言
4. 英文

### URL 策略

- `/en/...`
- `/es/...`
- `/pt-br/...`
- `/zh-cn/...`
- `/ja/...`
- `/ko/...`

每种语言有独立可索引 URL，并设置 `hreflang`。

切换语言时保持当前页面、比赛和审判不变，只切换内容语言。

### 翻译规则

- 固定界面文案使用人工确认过的核心词库和自动化回归测试。
- 实时审判问题和摘要由 AI 翻译。
- 球员和国家名称使用标准映射表，不让 AI 自由翻译。
- 语气保持短、直接、适合投票。
- 不逐字翻译梗或文化表达，允许本地化改写，但不得改变事实。
- 每个问题先生成“语言无关事实包”，再分别创作各语言版本，避免多次翻译造成事实漂移。
- 自动执行实体、数字、比分、分钟、选项方向和敏感词一致性校验。

### 语言质量分层

#### Tier A：品牌重点语言

`en`、`es`、`pt-BR`、`zh-CN`、`ja`、`ko`、`ar`

- 完整网站体验
- 完整人格和榜单文案
- 自动社交媒体内容
- 文化语气本地化

#### Tier B：完整产品语言

其余 11 种语言：

- 完整网站体验
- 完整投票和分享卡
- 社交媒体内容按比赛和热度动态启用

### 自动语言选择

每个比赛事件不需要向所有语言发布社交内容。系统根据以下因素选择 2 至 6 种重点语言：

- 参赛球队所在国家和主要语言
- 该语言站内实时活跃用户
- 该语言过去内容的分享和点击表现
- 当前地区时区
- 事件是否具有全球性
- 对应平台是否有已授权账号

## 6. AI 全自动内容系统

## 6.1 自动化目标

系统无需运营人员逐条填写内容。比赛数据进入后，AI 自动判断是否值得创建审判，生成四语言内容，完成安全和事实校验，然后发布。

## 6.2 数据输入

建议使用一个主体育数据供应商和一个备用供应商。

主数据需要覆盖：

- 赛程
- 比分
- 比赛状态
- 阵容和球员
- 进球、助攻
- 红黄牌
- 点球
- 换人
- VAR 或关键事件（若供应商提供）
- 比赛统计

不建议浏览器直接调用第三方体育 API。由服务端定时获取、缓存并统一转换为内部数据结构。

## 6.3 Global Buzz Radar / 全球爆点雷达

爆点雷达是产品的核心入口。它不只监听比分数据，而是持续监测全球足球相关信号，发现正在快速升温的话题。

### 信号来源

#### 一级：结构化事实源

- 体育数据主供应商
- 体育数据备用供应商
- 官方赛程、比赛报告和纪律公告
- 球队、足协、球员和教练的已验证官方账号

一级信号可以触发事实型话题。

#### 二级：授权或公开新闻发现源

- 全球新闻聚合 API
- GDELT 等全球新闻事件索引
- 媒体 RSS 和公开新闻页面
- 多语言体育媒体标题和摘要

二级信号用于发现话题和交叉验证。网站只保存事实摘要、标题、发布时间和原文链接，不复制整篇新闻。

#### 三级：社交热度信号

- X 关键词和账号信号（取决于 API 权限）
- YouTube 搜索和视频热度
- Reddit 公开社区信号（遵守社区和 API 规则）
- Google Trends（获得官方 API alpha 权限时）
- 站内搜索、投票、分享和二维码回流异常增长

三级信号只能证明“有人正在讨论”，不能单独证明事件为真。

### 监测对象

系统自动维护以下实体词典：

- 赛事名称和常见别名
- 国家队和多语言名称
- 球员、教练、裁判
- 城市、球场和比赛编号
- 常见争议事件词
- 拼写错误、昵称和社交媒体标签

实体词典每天自动从阵容、赛程和可信来源更新。

### 热度异常检测

系统以 5 分钟、15 分钟和 60 分钟窗口计算：

- 提及量增长速度
- 新来源数量
- 国家和语言覆盖范围
- 关键词共现
- 情绪变化
- 站内搜索增长
- 同一人物或事件的集中出现
- 是否与当前比赛时间线吻合

重点不是绝对提及量，而是相对平时的异常增长。小球队突然出现 20 倍增长，也可能比热门球队正常增长更值得关注。

### 事件聚类

不同语言、不同媒体对同一事件的描述必须合并成一个 `Story Cluster`：

```text
多源信号
→ 语言识别
→ 实体提取
→ 时间和比赛匹配
→ 相似事件聚类
→ 形成一个统一爆点候选
```

同一事件不能因为语言不同生成 18 个重复话题页。

### 爆点候选状态

| 状态 | 含义 | 系统动作 |
|---|---|---|
| `detected` | 检测到异常 | 继续抓取和聚类，不公开 |
| `developing` | 多个信号出现 | 准备问题和素材，不陈述为事实 |
| `verified` | 达到事实验证规则 | 自动创建话题页和投票 |
| `viral` | 热度和站内参与快速增长 | 多语言、多平台自动发布 |
| `resolved` | 官方结果或后续事实明确 | 更新结果并生成反转内容 |
| `retracted` | 原信息被否定 | 停止传播并发布更正 |

### 自动事实验证

满足以下任一条件才可进入 `verified`：

1. 体育数据源或赛事官方来源直接确认。
2. 一个官方已验证账号直接确认。
3. 两个彼此独立的可信新闻来源确认，且关键实体、时间和事件一致。

只有社交媒体传言时，系统不能写成事实。可以暂不公开，或在风险较低时生成明确标注的讨论题：

> Reports are emerging, but this has not been officially confirmed. If true, should the player be suspended?

涉及伤亡、犯罪、兴奋剂、歧视、政治、性侵或严重指控时，不允许基于传言自动创建娱乐化投票。

### 自动创建话题页

爆点进入 `verified` 后，系统自动生成：

- 一个统一话题 ID
- 多语言标题和简短事实摘要
- 来源与更新时间
- 相关比赛、球队和人物
- 一个主审判问题
- 1 至 3 个后续问题
- 可验证的投票选项
- 分享图片、动态图和短视频
- SEO 元数据和结构化数据
- 社交发布计划

URL 示例：

`/en/story/late-var-penalty-brazil-morocco`

### 话题页结构

1. `BREAKING / DEVELOPING / VERIFIED / RESOLVED` 状态
2. 一句话说明发生了什么
3. 来源确认时间
4. 立即投票
5. 全球和国家观点
6. 事件时间线
7. 来源链接
8. 自动更正记录
9. 分享卡和大二维码

系统不能隐藏后续更正。事实发生变化时，原 URL 保留并展示清晰更新记录。

## 6.4 内容生成触发器

| 事件 | 自动生成内容 |
|---|---|
| 开赛前 6 小时 | 赛前立场问题 |
| 开赛前 30 分钟 | 阵容相关问题 |
| 进球 | 英雄、责任或比赛走势问题 |
| 点球判罚 | 是否合理、谁承担压力 |
| 红牌 | 判罚是否过重、是否毁掉比赛 |
| VAR | 判罚和流程争议 |
| 错失点球 | 是否应该背锅 |
| 关键换人 | 教练判断 |
| 中场 | 上半场 Saint / Villain |
| 终场 | 本场 Saint / Villain |
| 终场后 30 分钟 | AI 结果摘要和国家差异 |

## 6.5 发布流水线

```text
体育数据、新闻和社交信号进入
→ 多语言实体识别和事件聚类
→ 事实验证
→ 事件重要性与 Viral Score
→ 查重
→ AI 生成候选问题与选项
→ 事实约束校验
→ 内容安全校验
→ 四语言生成
→ 反向翻译/实体一致性检查
→ 自动发布
→ 监控投票质量
→ 自动关闭或合并异常内容
```

## 6.6 事件重要性评分

只有达到阈值的事件才生成实时审判，避免信息过载。

评分因素：

- 是否改变比分
- 是否为点球、红牌或 VAR
- 比赛阶段
- 是否为淘汰赛
- 双方实力和比分差
- 数据源是否提供足够事实
- 最近是否已出现类似问题
- 当前页面是否已有过多未完成投票

首页同一时间最多突出 3 个实时审判。其余进入比赛详情页。

## 6.7 AI 输出结构

AI 必须输出结构化 JSON，不允许直接输出自由文本到页面：

```json
{
  "event_id": "string",
  "poll_type": "penalty_decision",
  "question": {
    "en": "Was that really a penalty?",
    "zh-CN": "这真的是点球吗？",
    "ja": "本当にPKだった？",
    "ko": "정말 페널티킥이었을까?"
  },
  "options": [
    {
      "id": "yes",
      "labels": {
        "en": "Clear penalty",
        "zh-CN": "明确是点球",
        "ja": "明らかにPK",
        "ko": "명백한 페널티킥"
      }
    }
  ],
  "fact_refs": ["provider_event_id"],
  "confidence": 0.96,
  "expires_at": "ISO-8601"
}
```

## 6.8 自动质量控制

以下任一情况不得自动发布：

- 数据源之间比分或事件冲突
- 无法确认球员身份
- AI 输出出现数据源中不存在的事实
- 爆点仅来自单一匿名社交账号或无法追溯的截图
- 问题与最近内容高度重复
- 使用侮辱性、歧视性或煽动性语言
- 将球员描述为犯罪者、作弊者等事实性指控
- 翻译后的选项含义不一致
- 置信度低于设定阈值

### 降级策略

1. 无法生成创意问题时，使用经过审核的事件模板。
2. 翻译失败时保留英文并标注“翻译处理中”，后台自动重试。
3. 主数据源异常时切换备用源。
4. 两个数据源冲突时暂停该事件投票。
5. AI 服务异常时仍保留赛程、已有投票和模板化赛后评选。

## 6.9 爆点识别引擎

系统为每个事件计算 `Viral Score`，决定是否进入首页、是否生成短视频、是否自动发社交媒体。

### Viral Score 因素

- 事件强度：绝杀、红牌、点球、VAR、门将失误、淘汰
- 比赛重要性：淘汰赛高于普通小组赛
- 球队关注度
- 观点分裂度：越接近五五开，讨论价值越高
- 结果反转速度
- 跨国家观点差异
- 单位时间投票增长
- 分享率和二维码回流率
- 是否存在可验证、简洁的冲突问题

### 自动动作

| Viral Score | 动作 |
|---|---|
| 0-39 | 只进入比赛详情页 |
| 40-59 | 进入首页事件流 |
| 60-79 | 生成分享卡并发布重点语言社交内容 |
| 80-100 | 进入爆点状态，生成多平台内容、短视频和国家对撞页 |

分数阈值根据实际转化自动调优，但不得为了提高分数虚构冲突。

## 7. 分享与二维码回流

## 7.1 分享原则

分享内容不能只是网站截图。它必须表达“我选择了什么”，让朋友产生赞同或反驳冲动。

### 分享触发点

- 完成投票
- 获得人格
- 进入少数派
- 国家排名变化
- 房间排名变化
- 每日 Saint / Villain 发布

## 7.2 分享卡规格

自动生成：

- 竖版动态/Story：1080 × 1920
- 社交信息流：1080 × 1350
- 横版链接预览：1200 × 630
- 方形：1080 × 1080

### 分享卡内容

- VileSaint 品牌
- 比赛双方和国旗
- 审判问题
- 用户选择
- 全球结果
- “你站在全球 X% 这一边”
- 大二维码
- 短链接
- `Scan to cast your verdict` 等本地化 CTA

## 7.3 大二维码规则

- 信息流卡二维码视觉尺寸建议 220 至 280px。
- Story 卡二维码建议 280 至 340px。
- 四周留出至少 4 个模块的静区。
- 使用高对比度深色码和浅色底。
- 二维码中心可以放简化的 `VS` 标识，但占比不得影响识别。
- 每张卡生成后自动进行二维码解码测试，测试失败不得提供下载。
- 二维码必须直接进入原审判页面，不进入首页。
- 页面自动识别分享来源，并在投票后回到相同分享闭环。

二维码 URL 示例：

`https://vilesaint.com/en/court/was-that-a-penalty?ref=share&c=8F2K`

## 7.4 原生分享

优先使用 Web Share API。提供：

- Copy link
- Download image
- WhatsApp
- LINE
- KakaoTalk
- X
- Facebook
- 微信提示（下载图片或复制链接）

平台按钮根据用户语言和设备排序，不在所有地区固定展示全部平台。

## 7.5 好玩、有冲突、有爆点的互动机制

### A. World Split / 全球撕裂指数

当两个选项接近五五开时，显示动态仪表：

> The world is split: 51% vs 49%

用户能看到哪个国家正在把结果推向另一边。

### B. Nation Clash / 国家观点对撞

自动找出观点差异最大的两个国家：

> 82% of Brazil says YES. Only 19% of Japan agrees.

页面提供“代表你的国家投一票”，但必须注明结果仅代表本站参与用户。

### C. Receipt / 立场存证

赛前或事件刚发生时，用户可以锁定判断。之后系统生成：

- `You called it before 91% of fans`
- `You changed your verdict after the replay`
- `You stood alone. You were right.`

这是荣誉记录，不涉及金钱或博彩。

### D. Call Out a Friend / 点名挑战

分享时可以选择：

- “你敢不敢判？”
- “我们不可能对这件事意见一致”
- “证明我错了”

生成带双方空位的挑战卡。朋友扫码投票后，展示两人的观点是否一致。

### E. Verdict Flip / 舆论反转

记录投票比例随时间变化。当多数意见反转时自动生成事件：

> The verdict just flipped.

适合发布动态图或 6 至 10 秒短视频。

### F. Halo vs Horns / 光环与恶角

用户头像和人格卡根据近期判断获得动态 Saint/Villain 外观。不得使用宗教侮辱元素，表现为抽象光环和尖角图形。

### G. Last Fan Standing / 最后坚守者

当用户一直坚持少数观点并最终被比赛结果或后续官方决定支持时，获得限时徽章。

### H. No Comment, Just React

不开放公共自由评论，但提供预设反应：

- Robbery
- Fair call
- Pure chaos
- Cold-blooded
- Heartbreaker
- Cinema

反应按语言本地化，自动形成实时情绪条。

## 8. 零推广预算增长设计

零推广预算意味着产品自身必须产生流量入口。

## 8.1 实时事件增长

- 每个争议事件都有独立 URL。
- 事件发生后尽快生成页面。
- 标题包含比赛、人物和问题，但不堆砌关键词。
- 页面包含结构化比赛信息和实时投票结果。

## 8.2 程序化 SEO

自动生成并持续更新：

- 每场比赛页
- 每支国家队页
- 每位入选球员页
- 每个 Saint / Villain 榜单页
- 每日争议汇总页
- 四语言版本

禁止生成内容稀薄、彼此重复的页面。没有独立事实或用户数据的页面不进入索引。

## 8.3 分享回流

每个分享卡必须包含：

- 明确立场
- 可争论的问题
- 当前结果
- 大二维码
- 深链接

目标是让接收者不需要解释就能扫码反驳或支持。

## 8.4 好友房间增长

房间页面鼓励成员邀请其他人，但不使用骚扰式弹窗。每新增 3 名参与者，自动生成一次房间观点总结，形成新的可分享内容。

## 8.5 长尾持续性

网站架构不能只适用于本届赛事。赛事结束后可以无缝覆盖：

- 国家队比赛
- 欧洲杯、美洲杯、亚洲杯
- Champions League
- 各国联赛

品牌始终围绕足球观点审判，不绑定单一赛事名称。

## 8.6 自动社交媒体发布引擎

### 目标

将网站中的实时事件自动转化为平台原生内容，而不是只发链接。系统完成选题、语言选择、创作、渲染、发布、效果追踪和后续改写。

### 首选平台

| 优先级 | 平台 | 自动内容 |
|---|---|---|
| P0 | X | 快速争议问题、实时比例、舆论反转、图片 |
| P0 | Instagram | 结果卡、Carousel、Reels、Stories |
| P0 | Facebook | 图片投票、国家观点对撞、链接 |
| P0 | TikTok | 6-15 秒动态投票短视频 |
| P1 | Threads | 短问题、实时结果、讨论链接 |
| P1 | YouTube Shorts | 10-20 秒榜单和反转短视频 |
| P1 | Bluesky | 英文实时争议和结果卡 |
| P1 | Mastodon | 英文实时争议和结果卡 |
| 限制 | Reddit | 不进行跨社区自动灌水；仅在自有社区或获得版主许可时发布 |

平台发布依赖官方 API、账号授权、应用审核和平台权限。某平台没有批准自动发布权限时，系统应降级为自动生成草稿，不使用浏览器脚本模拟人工发帖。

### 社交账号矩阵

不建议一开始创建 18 套全平台账号。首发采用：

- `VileSaint Global`：英文，覆盖全球事件
- `VileSaint Español`
- `VileSaint Brasil`
- `VileSaint 日本`
- `VileSaint 한국`
- `VileSaint 中文`
- `VileSaint عربي`

其他语言根据站内自然流量和球队比赛动态发布到 Global 账号，或后续自动扩展独立账号。

### 自动发布流程

```text
爆点雷达发现候选
→ 聚类与事实验证
→ 自动创建网站话题页
→ Viral Score
→ 选择平台与语言
→ 生成平台原生文案
→ 生成图片/Carousel/短视频
→ 事实与安全校验
→ 检查发布频率和重复度
→ 官方 API 发布
→ 写回 post_id
→ 追踪点击、投票、分享和二维码回流
→ 根据效果调整下一次内容
```

网站必须先生成可追溯、可更新的话题页，再向社交媒体发布。社交媒体是分发层，VileSaint 网站是事实、投票、更新和回流的唯一主页面。

### 爆点自动传播闭环

```text
全球信号出现
→ 系统发现异常增长
→ 多源合并成一个事件
→ 自动验证事实
→ 自动生成多语言话题和投票
→ 自动生成图片、动态图、短视频
→ 自动发布到最相关的语言账号和平台
→ 用户点击或扫码进入话题页
→ 投票结果形成新的爆点
→ 自动发布“全球分裂 / 国家对撞 / 舆论反转”
→ 更多用户回流
```

### 平台内容模板

#### X

- 首句直接提出冲突。
- 单条只表达一个问题。
- 优先使用实时比例和深链接。
- 不使用大量标签。

示例：

> 51% say penalty. 49% say robbery.  
> The world cannot agree. Cast your verdict:

#### Instagram

- 第一张：问题和冲突
- 第二张：全球结果
- 第三张：国家差异
- 第四张：大二维码和 CTA
- Reels 使用动态百分比、国旗对撞和 6 至 10 秒节奏

#### TikTok / Shorts

```text
0-2 秒：争议事件问题
2-5 秒：实时比例快速变化
5-8 秒：两个国家观点对撞
8-12 秒：大二维码 + Cast your verdict
```

视频只使用文字、数据、国旗、品牌图形和获授权素材，不使用未授权比赛视频。

#### Facebook

- 使用问题式文案
- 展示一张结果卡
- 链接进入对应语言审判页
- 鼓励用户投票，不鼓励在评论区攻击球员或国家

### 发布时机

- 重大事件确认后 1 至 3 分钟：X / Threads / Bluesky
- 获得首批有效投票后：更新结果帖
- 比赛结束后 5 至 10 分钟：Saint / Villain 卡
- 观点反转发生时：自动发布 Verdict Flip
- 每个地区晚间黄金时间：当天榜单和人格内容

不能为了速度绕过数据确认。单一数据源尚未确认的争议事件只能发布问题，不得陈述为确定事实。

### 频率控制

- 同一平台、同一语言、同一比赛不得连续发布高度相似内容。
- 低价值事件不发社交媒体。
- 大赛期间仍设置每小时上限。
- 连续低点击内容自动降低频率。
- 高回流内容可以生成后续结果帖，但最多追加一次。

### 自动效果学习

每条社交内容记录：

- 平台和语言
- 内容类型
- 发布时间
- 事件类型
- 展现、互动和点击（平台允许时）
- 深链接访问
- 新增投票
- 二维码回流
- 新用户和房间创建

优化目标不是点赞数，而是：

> 每 1,000 次社交曝光带来的有效投票数。

### 自动发布安全阀

- 账号授权失效时暂停该账号，不重复失败刷接口。
- 平台 API 限流时进入队列并判断内容是否已过时。
- 内容过时后自动丢弃，不延迟发布“实时”新闻。
- 重大伤病、暴力、歧视或政治敏感事件不自动娱乐化。
- 自动发布系统必须支持全局停止开关。

## 9. 视觉设计规范

## 9.1 视觉方向

关键词：

- 极简
- 高冲突
- 编辑感
- 法庭感
- 全球化
- 移动端优先

避免：

- 传统体育数据网站的密集表格
- 博彩网站式霓虹绿
- 大量渐变和玻璃拟态
- 复杂 3D 足球元素
- 官方赛事视觉的模仿

## 9.2 色彩

- `Ink`：#111111，主背景和正文
- `Paper`：#F5F1E8，浅色背景
- `Saint`：#F2C94C，圣徒金
- `Villain`：#D9363E，反派红
- `Verdict`：#3457D5，交互蓝
- `Muted`：#8B887F，辅助文本
- `Success`：#2E8B57

默认采用浅色纸张背景与黑色文字。关键直播和 Villain 状态使用红色，Saint 使用金色。颜色不能成为表达含义的唯一方式。

## 9.3 字体

优先使用系统字体，避免首屏等待大型字体文件：

- English：Inter / system-ui

## 12. 技术架构建议

## 12.1 推荐技术栈

- 前端：Next.js + TypeScript
- 样式：Tailwind CSS
- 数据库：PostgreSQL
- 鉴权：Supabase Auth 或兼容服务
- 实时结果：Postgres Realtime / WebSocket
- 缓存与限流：Cloudflare KV / Redis
- 定时任务：Cloudflare Workers Cron 或托管任务队列
- 图片生成：服务端 HTML/CSS 模板转图片
- 视频生成：服务端模板化动态图渲染（Remotion 或同类方案）
- 二维码：服务端生成并自动解码验证
- AI：支持结构化 JSON 输出的模型 API
- 社交发布：各平台官方发布 API + OAuth 账号授权
- 发布队列：支持延迟、去重、过时丢弃和失败重试的任务队列
- 部署：Cloudflare + Vercel 或同等级全球 CDN
- 监控：Sentry + 结构化日志 + 可用性探测

## 12.2 高层架构

```text
用户浏览器
  ↓
全球 CDN / WAF
  ↓
Next.js Web 应用
  ↓
应用 API
  ├─ PostgreSQL
  ├─ 实时投票结果
  ├─ 缓存与限流
  ├─ 分享图生成服务
  ├─ 短视频渲染服务
  ├─ 社交发布编排器
  └─ AI 内容管线
        ↑
体育数据主供应商
        ↑
体育数据备用供应商
```

## 12.3 架构决定

### ADR-001：采用模块化单体，而不是微服务

原因：开发和运维资源有限，需要快速上线。比赛同步、AI 管线和图片生成使用独立任务进程，但共享一套代码和数据库模型。

### ADR-002：AI 不直接写入已发布内容

原因：防止幻觉和翻译错误。AI 输出先进入结构化校验层，只有满足事实、安全和置信度规则才发布。

### ADR-003：投票聚合采用缓存

原因：比赛争议发生时会产生尖峰流量。写入数据库后使用增量计数和短期缓存返回结果，避免每次请求扫描全部票数。

### ADR-004：分享图片按需生成并缓存

原因：相同审判和结果区间可复用模板，减少图片生成成本。个人选择、语言和来源码组成缓存键。

### ADR-005：社交媒体只通过官方 API 发布

原因：浏览器模拟发布容易失效，也可能违反平台规则。官方 API 虽然需要账号授权、应用审核或产生使用成本，但可审计、可重试、风险更低。

### ADR-006：站内语言与社交语言分离

原因：网站可同时支持 18 种语言，但同一事件不应向所有语言账号机械群发。社交编排器按球队、地区、热度和历史效果选择语言。

## 14. 内容安全与防滥用

### 14.1 投票内容

AI 问题可以尖锐，但不能：

- 使用种族、国籍、性别或宗教攻击
- 鼓励暴力
- 对球员家人进行评价
- 将争议行为描述为未经证实的犯罪事实
- 生成羞辱外貌或伤病的内容

### 14.2 用户输入

首发版本尽量减少自由文本：

- 房间名需要审核
- 昵称需要审核
- 不开放公共评论
- 使用预设反应代替自由评论

### 14.3 反作弊

- 会话、账户、设备和短期 IP 哈希组合限流
- 异常投票速率检测
- 相同来源批量投票隔离
- 排行榜排除高风险票，但原用户仍看到提交成功，避免对抗升级
- 公开显示“有效投票数”，不保证等于原始请求数

- 重复问题比例

- 各语言内容的异常低表现
- 爆点候选发现延迟
- 事件聚类准确率
- 单一来源误触发数量

紧急能力：

- 全局暂停新的 AI 内容发布

这些功能用于安全和故障处理，不用于日常人工填内容。

## 18. 完整上线验收标准

### 核心功能

- 用户无需登录即可完成投票。
- 投票后显示全球、国家和语言群体结果。
- 比赛数据能够自动同步。
- 爆点雷达能够自动监测体育数据、可信新闻和允许使用的社交热度信号。
- 多语言来源对同一事件能够聚合成一个话题。
- 未达到验证规则的传言不会被自动发布为事实。
- 通过验证的爆点能够自动创建话题页、投票、来源列表和更新记录。
- 关键事件能够自动创建审判。
- 比赛结束后自动生成 Saint / Villain 评选。
- 18 种语言可以在任何页面无损切换。
- AI 内容通过结构化校验后自动发布。
- 用户完成足够投票后自动生成人格。
- 国家榜、人物榜和争议榜自动更新。
- 用户能够创建并通过链接或二维码加入好友房间。
- 高 Viral Score 事件能自动生成平台原生内容并通过已授权官方 API 发布。
- 自动发布内容能够追踪至站内投票和二维码回流。
- 全球撕裂、国家对撞、立场存证和舆论反转机制能够自动触发。

### 分享

- 每个核心结果都能生成 4 种尺寸的分享卡。
- 分享卡包含清晰的大二维码。
- 所有二维码通过自动解码测试。
- 扫码直接进入对应语言和对应审判。
- 分享来源能够被统计。

### 质量

- 不使用官方赛事标识和未授权比赛图片。
- AI 不生成数据源中不存在的比赛事实。
- AI 服务故障时网站仍可浏览和投票。
- 热点流量下投票不重复、不丢失。
- 手机端单手可以完成全部核心流程。
- 键盘和屏幕阅读器可以完成投票。

## 20. 最终产品形态

VileSaint 不是用户偶尔查询一次的工具，而是一场比赛期间反复打开的“观点现场”：

```text
比赛制造事件
→ AI 自动提出值得争论的问题
→ 用户快速站队
→ 全球结果强化身份与冲突
→ 分享卡表达个人立场
→ 大二维码带回新用户
→ 好友房间和人格系统形成留存
```

判断产品是否成功，不看它收录了多少足球资料，而看每当比赛出现争议时，用户是否会自然想到：

> 去 VileSaint 看看全世界怎么判。
